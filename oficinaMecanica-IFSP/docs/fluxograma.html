📡 Fluxo de uma requisição autenticada

  1. Cliente envia Authorization: Bearer <token>

    Header obrigatório em todos os endpoints, exceto POST /auth/login

  2. JwtAuthenticationFilter valida o JWT

    Verifica assinatura HS256, expiração e extrai email + role

  3. Spring Security popula o SecurityContext

    Usuário autenticado disponível via Authentication para @PreAuthorize

  4. Controller recebe a requisição e chama o Service

    Bean validation (@Valid) ocorre antes de chegar ao método

  5. Service aplica as regras de negócio e persiste via Repository

    Exceções de negócio → 422, Not Found → 404

  6. Controller serializa a resposta para JSON e retorna ao cliente

💡 Stateless por design

Não há sessão no servidor. Cada requisição precisa do JWT completo. O token carrega email, role e exp (expiração) no payload. Segredo configurado via variável de ambiente JWT_SECRET.

📸 Snapshot Pattern

Quando um produto ou serviço é adicionado a uma OS, seus dados (nome, preço) são copiados — snapshot — para OrderProduto / OrderServico. Mudanças futuras no catálogo não afetam OSs já abertas.

💡 Total calculado automaticamente

Order.getTotal() soma OrderItem.getSubTotal() de cada item.
Produto: preço × quantidade
Serviço: preço × 1

⚠️ Remoção devolve ao estoque

Ao remover um produto da OS via DELETE /orders/{id}/produtos/{produtoId}, o estoque é incrementado automaticamente pela quantidade que foi reservada.

📋 Geração de orçamento

Ao mover o status para AGUARDANDO_APROVACAO, o campo dataValidade é obrigatório no body. Isso gera o orçamento com prazo definido. O total da OS no momento é o valor do orçamento (Σ serviços + Σ peças × quantidade).

🔁 Respostas de erro padronizadas

401 Unauthorized — token ausente, inválido ou expirado
403 Forbidden — autenticado mas sem permissão para o endpoint
404 Not Found — entidade não encontrada no banco
400 Bad Request — falha de validação (@NotBlank, @Positive, @Email…)
422 Unprocessable Entity — violação de regra de negócio
